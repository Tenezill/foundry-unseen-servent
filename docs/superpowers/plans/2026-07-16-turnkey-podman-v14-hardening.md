# Turnkey Podman + v14 Hardening (Phase 1) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `make setup` produce a working stack on rootless Podman (fix the felddy `/data` + secret permission failures) and bump Foundry from v13.351 to v14, with no other version changes.

**Architecture:** felddy `foundry` is the only service that runs as a fixed non-root uid, so it's the only one that breaks under rootless Podman's host-user→container-root mapping. `make setup` detects the runtime and, on the Podman path only, writes a `docker-compose.override.yml` giving `foundry` `userns_mode: "keep-id"` (container-uid == host-uid). The v14 bump is tag-only because module 3.4.1, relay 3.4.1, and dnd5e 5.3.3 are all already v14-verified.

**Tech Stack:** Node 22 ESM script (`scripts/setup-quickstart.mjs`), Vitest, Docker/Podman Compose, felddy/foundryvtt image.

## Global Constraints

- Node >= 22 (`package.json` engines); pnpm 11.10.0.
- **Never pin `:latest` or a rolling major tag** for any image (`VERSIONS.md` policy). Foundry pins to an exact build.
- Foundry pin target: **`felddy/foundryvtt:14.364.0`** (felddy Docker Hub stable). Confirm it is felddy's current stable tag at build time; if a newer stable exists tracking core ≤ 14.365, pin that exact build instead.
- **`keep-id` goes on `foundry` ONLY.** Applying it to root-running services (`relay`, `gateway`, `web`, `bootstrap`) breaks them (their container-root must keep mapping to the host user).
- Generated files (`.env`, `Caddyfile.tls`, `docker-compose.override.yml`, `secrets/*`) are never committed and are wiped by `make setup-reset`.
- Existing test style: pure functions in `scripts/setup-quickstart.mjs` are exported and unit-tested in `apps/bootstrap/test/setup-cli.test.ts`. `main()` is not unit-tested.

```js
export const PODMAN_OVERRIDE_MARKER = '# generated by `make setup` (rootless-Podman)';

export function isPodmanRuntime(compose) {
  return Array.isArray(compose) && compose[0].startsWith('podman');
}

// felddy foundry runs as a fixed non-root uid; under rootless Podman the host
// user maps to container-root, so foundry (uid 1000) cannot read/write the
// host-owned bind mount or the 0600 secret. keep-id maps its uid to the host
// user, fixing both — and keeping the secret readable by the host user too.
// ONLY foundry needs this: relay/gateway/web/bootstrap run as root (or start as
// root and self-chown), which already maps to the host user under rootless Podman.
export function buildPodmanComposeOverride() {
  return (
    `${PODMAN_OVERRIDE_MARKER} — do not edit; auto-removed on the docker path.\n` +
    'services:\n' +
    '  foundry:\n' +
    '    userns_mode: "keep-id"\n'
  );
}
```

```js
    const compose = detectComposeCommand();

    // Rootless-Podman ownership fix: give foundry keep-id via a generated
    // override (Podman only). Managed even under --no-up so a later manual
    // `podman compose up` still gets it. Removed on the docker path, but only
    // if we generated it (never clobber a user-authored override).
    const override = join(QDIR, 'docker-compose.override.yml');
    if (isPodmanRuntime(compose)) {
      writeFileSync(override, buildPodmanComposeOverride(), 'utf8');
      console.log('rootless-Podman detected → wrote docker-compose.override.yml (foundry keep-id).');
    } else if (existsSync(override) && readFileSync(override, 'utf8').startsWith(PODMAN_OVERRIDE_MARKER)) {
      rmSync(override, { force: true });
    }

    if (args.includes('--no-up')) return;
    if (compose === null) {
      console.error('no container runtime found — install docker (with compose v2) or podman.');
      process.exitCode = 1;
      return;
    }
    console.log(`\nrunning: ${compose.join(' ')} up -d --build   (in stack/quickstart)`);
    const up = spawnSync(compose[0], [...compose.slice(1), 'up', '-d', '--build'], { cwd: QDIR, stdio: 'inherit' });
    process.exitCode = up.status ?? 1;
```

```markdown
#### Rootless-Podman file ownership

`make setup` detects Podman and writes `stack/quickstart/docker-compose.override.yml`
giving the `foundry` service `userns_mode: "keep-id"`. This is required because
felddy runs as a fixed non-root uid, which under rootless Podman cannot read the
host-owned `foundry_data` bind mount or the `0600` `secrets/foundry-config.json`.
`keep-id` maps foundry's uid to your host user, so both work — and you keep read
access to your own admin key. The override is **not** applied on Docker (where it
is unnecessary and would misbehave) and is auto-removed if you switch to Docker.
Only `foundry` gets it: `relay`, `gateway`, `web`, and `bootstrap` run as root
(or start as root and self-chown), which already maps to the host user.
```
