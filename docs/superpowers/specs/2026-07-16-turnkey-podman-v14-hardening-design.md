# Turnkey setup hardening — Phase 1: rootless-Podman ownership + Foundry v14

**Date:** 2026-07-16
**Status:** Design (Phase 1 of 3)
**Scope:** `stack/quickstart/` turnkey stack + `scripts/setup-quickstart.mjs`

## Problem

`make setup` fails to produce a working stack on **rootless Podman** (the felddy
foundry container crash-loops on `insufficient permissions on /data` and cannot
read its mounted secret file), and the stack is pinned to **Foundry 13.351**,
which is a major version behind current stable **v14**.

Both problems surfaced during a real turnkey deploy on a rootless-Podman host.
The permission failures were *misdiagnosed as bad credentials* because felddy's
"set FOUNDRY_USERNAME…" message is emitted when it cannot **read** the secret
file, not only when credentials are wrong.

This spec covers **Phase 1 only** — the changes that get a clean, working v14
stack on both Docker and rootless Podman. Phases 2–3 (setup-script UX) are
listed under *Future phases* and will get their own specs.

## Root-cause analysis (verified)

### Ownership
Under rootless Podman, the host user (`uid 1000`) maps to **container root
(uid 0)**. Bind-mount folders and the `0600` secret files are created owned by
the host user, so *inside* a container they appear owned by root.

- `gateway`, `web`, `bootstrap` have **no `USER` directive → run as root** →
  root == host user → they can read/write host-owned bind mounts. **Already
  correct, no change needed.** (`bootstrap` additionally chowns/drops to uid
  3000 via its own entrypoint — unchanged.)
- `foundry` (felddy) runs as a **fixed `uid 1000`**, which maps to a *subuid*,
  not the host user → it cannot read/write host-owned `./foundry_data` or the
  `0600` `foundry-config.json` secret. **This is the sole broken service.**

### Version compatibility (verified against upstream, 2026-07-16)
A v14 bump is a **tag-only** change — every downstream pin is already v14-ready:

| Component | Current pin | v14 status |
|---|---|---|
| foundry-rest-api **module** | 3.4.1 | Verified against v14 (since 3.1.0) — no change |
| **relay** | 3.4.1 | Matches module, v14-ready — no change |
| **dnd5e** system | 5.3.3 | 5.3 line supports v13 **and** v14 — no change |
| adapter fixtures | dnd5e 5.3.3 | Unchanged → **fixtures untouched** |
| Foundry core | **13.351.0** | Bump → **14.364.0** (felddy Docker Hub stable) |

## Design

### A. Rootless-Podman ownership fix

Apply **`userns_mode: "keep-id"` to the `foundry` service only**, so its
container-uid-1000 becomes identical to the host user. Then the host-owned
`./foundry_data` and the `0600` `foundry-config.json` are read/writable by both
the container **and** the host user (the latter matters for Phase 2's
`make show-secrets` / admin-key retrieval).

**Do NOT blanket `keep-id`.** It would remap container-root away from the host
user and break the root-running services (`gateway`, `web`, `bootstrap`).

**Docker parity.** `keep-id` is Podman-only. `make setup` already detects the
runtime (`detectComposeCommand`). The fix:

1. Extend detection to distinguish **podman vs docker** (return the runtime
   family, not just the command array).
2. On the **podman** path, generate
   `stack/quickstart/docker-compose.override.yml`:
   ```yaml
   # generated by make setup on the rootless-Podman path — do not edit
   services:
     foundry:
       userns_mode: "keep-id"
   ```
   Compose auto-loads `docker-compose.override.yml` from the project dir, so no
   flag changes are needed. Docker users never get this file.
3. On the **docker** path, ensure any stale override is removed (so switching
   runtimes doesn't leave a Docker-incompatible file behind).
4. Treat the override like `.env`: add to `.gitignore` and remove it in
   `make setup-reset`.

**`relay` uid:** verify during implementation (`podman inspect` /
image `Config.User`). If `relay` also runs non-root, add it to the override.
If it runs as root (expected), leave it out.

### B. Foundry v13 → v14

- Replace `felddy/foundryvtt:13.351.0` → `felddy/foundryvtt:14.364.0` in:
  - `stack/docker-compose.dev.yml`
  - `stack/docker-compose.prod.yml`
  - `stack/quickstart/docker-compose.yml`
- Update the Foundry row in `VERSIONS.md` (13.351 → 14.364).
- Confirm `14.364.0` is felddy's current Docker Hub **stable** tag at
  implementation time; if felddy has published a newer stable that still tracks
  core ≤ 14.365, pin that exact build instead (never `:14` or `:latest` —
  per `VERSIONS.md` policy).
- **No** change to dnd5e, module, relay, or adapter fixtures.

### C. Documentation
- `docs/HOSTING.md` Part C: document the rootless-Podman ownership model (why
  only `foundry` needs `keep-id`, that the override is auto-generated) and the
  v14 pin.
- Remove/replace any manual `podman unshare chown` guidance if present.

## Verification (per VERSIONS.md upgrade policy)

1. **Fixture tests** (`pnpm -r test`) — unchanged dnd5e pin, must stay green.
2. **Clean rootless-Podman deploy** from scratch:
   - Reset stale ownership from prior manual chowns:
     `podman unshare chown -R 0:0 stack/quickstart/{foundry_data,relay-data,gateway-data,caddy-data,secrets}`
   - `make setup-reset` then `make setup` (or `podman compose up -d`).
   - Assert: `foundry` reaches `listening on port 30000` (no volume-test or
     jq permission errors); host user can `cat secrets/foundry-config.json`.
3. **Live round-trip on v14:** create a world, enable the REST API module, one
   read/write/SSE round-trip through the relay (the `docs/M0-findings.md`
   expectation).
4. **Docker parity smoke:** confirm no `docker-compose.override.yml` is written
   on the docker path and compose still validates.

## Out of scope / future phases

- **Phase 2 (setup UX):** move "Next steps" to after `compose up`; add
  `make show-secrets`.
- **Phase 3 (credential validation):** log-watching re-prompt loop that
  distinguishes *rejected* vs *unreadable* vs *still-working* vs *success*, and
  overwrites only the foundry credentials on failure (generated secrets
  preserved), validating every run.

## Rollout note (this user's live host)

Because prior manual chowns left mixed ownership, the redeploy must first reset
ownership to the host user (step 2 above) so `keep-id` + default mappings start
from a clean state.
